## 二、产品背景

### 2.1 市场现状
**现有机器人能力空白：**
1. **跨市场语义理解能力为零** - 全是数字驱动，无法理解市场间语义关联
2. **逻辑链定价矛盾识别缺失** - 无法识别"AFC赢得超级碗"(45%) < "酋长队赢得超级碗"(50%)这类矛盾
3. **跨平台套利基础设施不完整** - 竞争对手显示"Kalshi即将支持"

### 2.2 产品定位
**不做：** 毫秒级数学套利（红海竞争）  
**做：** LLM驱动的语义理解 + 行为预测（蓝海机会）

### 2.3 核心价值主张
> "发现别人看不到的机会，跟随聪明钱，数据驱动决策"

### 3.2 模块详细需求

#### 模块 1: 语义套利系统 🔴 P0

**功能描述**  
使用 LLM 识别语义相关的预测市场，检测定价矛盾，生成套利信号。

**用户故事**
> 作为交易者，我希望系统能自动发现"AFC赢得超级碗"和"酋长队赢得超级碗"之间的定价矛盾，这样我就能抓住别人看不到的套利机会。

**功能需求**

| 功能ID | 功能名称 | 需求描述 | 验收标准 |
|--------|---------|---------|---------|
| FR-1.1 | 语义相似度计算 | 使用LLM计算两个市场的语义相似度 | 相似度>0.8的准确率>90% |
| FR-1.2 | 逻辑关系识别 | 识别implies/mutex/related三种关系 | 准确率>85% |
| FR-1.3 | 定价矛盾检测 | 检测违反逻辑关系的定价 | 准确率>95% |
| FR-1.4 | 套利信号生成 | 生成包含操作建议的信号 | 格式符合规范 |

**示例输出**
```
🔴 语义套利机会 detected!

关系类型: implies (AFC wins → Chiefs can win)
矛盾: AFC(45%) < Chiefs(50%)

套利操作:
- SELL Chiefs @ 50%
- BUY AFC @ 45%
- 预期收益: 5%

置信度: 95%
推理: Chiefs是AFC球队，如果Chiefs赢则AFC必然赢，
      因此Chiefs概率不应超过AFC概率。
```

**非功能需求**
- LLM调用延迟 <3秒
- 支持1000+市场/天
- API成本 <$200/月

#### 4.2.1 语义套利引擎

**架构说明**: 使用 OpenClaw 内置的 kimi-k2.5，采用任务队列模式

```python
class SemanticArbitrageEngine:
    """语义套利引擎"""
    
    def __init__(self):
        self.db = Database()
        self.cache = Cache()
    
    def calculate_semantic_similarity(self, market_a: Market, market_b: Market) -> Dict:
        """
        计算语义相似度
        
        Returns:
            包含prompt的字典，供外部LLM调用
        """
        prompt = f"""分析以下两个预测市场的语义相似度：

市场A: {market_a.title}
描述: {market_a.description}

市场B: {market_b.title}
描述: {market_b.description}

请给出相似度评分（0-1）和理由。
以JSON格式返回：{{"similarity": 0.85, "reason": "..."}}"""
        
        return {
            "type": "similarity",
            "prompt": prompt,
            "market_a": market_a.id,
            "market_b": market_b.id
        }
    
    def process_llm_result(self, task_type: str, response: str, **kwargs) -> Any:
        """处理LLM返回的结果"""
        result = json.loads(response)
        
        if task_type == "similarity":
            return float(result.get("similarity", 0))
        elif task_type == "relationship":
            return SemanticRelationship(
                relationship_type=result.get("relationship"),
                confidence=result.get("confidence")
            )
```

**LLM 调用流程**:
1. Python 程序生成 prompt，保存到任务文件
2. OpenClaw Agent 读取任务，调用 kimi-k2.5
3. OpenClaw Agent 保存结果
4. Python 程序读取结果，继续处理

## 五、界面设计规范

### 5.1 设计原则

#### 信息密度原则
- **关键信息前置**：最重要的决策信息在卡片第一屏展示
- **渐进式披露**：次要信息通过"查看详情"展开
- **视觉层次**：用颜色、大小、位置区分信息优先级

#### 交互效率原则
- **一键操作**：常用操作不超过2次点击
- **快捷预览**：悬停/展开即可查看详情，无需跳转页面
- **智能推荐**：根据用户行为推荐最相关的内容

### 5.5 交互规范

#### 悬停效果
- 卡片悬停: 轻微阴影 + 边框高亮
- 按钮悬停: 颜色加深
- 链接悬停: 下划线显示

#### 加载状态
- 列表加载: Skeleton 骨架屏
- 按钮加载: Spin 旋转图标
- 页面加载: 进度条或全屏 Loading

#### 空状态
- 图标 + 文字说明 + 操作引导
- 示例: "暂无信号，点击刷新查看最新数据"

### 5.6 响应式设计

#### 桌面端 (≥1200px)
- 三列布局: 鲸鱼列表 | 信号详情 | 操作面板
- 完整展示所有信息

#### 平板端 (768-1199px)
- 两列布局: 鲸鱼列表 | 信号详情
- 操作面板折叠到底部

#### 移动端 (<768px)
- 单列布局
- 卡片式展示
- 左右滑动切换标签
